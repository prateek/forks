# ghostpepper — fork learnings

Durable, fork-specific gotchas for building and syncing ghostpepper. Append here
whenever a build or sync failure teaches something that will recur. Cross-fork
lessons belong in the `fork-ops` skill's "Gotchas" instead.

## Build

- `GhostPepper/Secrets.swift` is gitignored. The build copies the committed
  `Secrets.example` into place first; without it xcodebuild fails with
  "cannot find 'Secrets' in scope".
- The Xcode project is generated by `xcodegen`, not committed. Generate before
  building; the build job `brew install`s xcodegen when it's missing.
- The Release build is memory-heavy. It runs `nice`d and `-jobs`-bounded under a
  short timeout so it can't wedge a shared mini. An earlier `@testable` Debug
  build was heavier still and OOM'd the host.
- Smoke is build-only (`test -d …/GhostPepper.app`), not the test suite: the
  suite needs network the secrets-free build job doesn't have.
- The built `.app` lands under `build/derived/Build/Products/Release/`. The
  release asset is tarred at its root by basename so the cask's `app "…"` finds
  it after extraction.

## Packaging

- The app ships ad-hoc signed — no Developer ID, no notarization. The cask strips
  quarantine in a `postflight` so it launches even on a Jamf-managed Mac, where
  Gatekeeper only blocks *quarantined* apps.
